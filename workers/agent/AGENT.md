# workers/agent

## Purpose

This directory contains the Treadmill agent worker, the execution substrate that polls the API for task assignments, orchestrates Claude Code within an isolated Git workspace, drives code changes through the repository, and publishes step lifecycle events back to the API. Each worker is an ECS task that exits after completing a single step, enabling stateless autoscaling driven by SQS queue depth. The worker implements the step-to-PR journey: fetch the task, reserve workspace isolation, run Claude Code, validate outputs, open/update the PR, and publish success or failure back to the API.

## Key surfaces

- `treadmill_agent/__main__.py` — entry point; orchestrates task fetch, workspace setup, Claude Code execution, PR interaction, and event publishing.
- `treadmill_agent/runner.py` — core step execution orchestrator; polls API for available tasks, manages the lifetime of a single step, coordinates dispositions (code, plan, validation, review, analysis).
- `treadmill_agent/runner_dispositions/` — pluggable step-type handlers: `code.py` (wf-author), `plan_doc.py` (wf-plan), `validation.py` (wf-validate), `review.py` (wf-review), `analysis.py` (wf-analyze).
- `treadmill_agent/workspace.py` — Git workspace lifecycle; creates isolated per-task directories, initializes repos, manages branch state, triggers Claude Code, captures outputs.
- `treadmill_agent/claude_code.py` — Wrapper that invokes the Claude Code CLI, parses its output, and routes to appropriate event channels.
- `treadmill_agent/startup_auth.py` — GitHub auth bootstrap. In `github_auth_mode='app'` the worker calls `bootstrap_github_auth_via_app` twice per lifecycle: once at startup (no repo → home-installation token) and again per task from `runner._handle_step` once `ctx.repo` is known (`repo='owner/name'` → token scoped to that repo's installation, so `gh` can clone / push outside the home installation). Also hosts `fetch_repo_worker_deps` (ADR-0059 step 2) — the per-step GET to `/api/v1/onboarding/repos/{repo}` that returns the repo's `WorkerDeps`; 404 / 503 / network failure returns `WorkerDeps()` (no overlay) so the legacy no-deps path stays in scope.
- `treadmill_agent/repo_deps.py` — Per-repo extras materialization (ADR-0059 step 2). `materialize(repo, worker_deps)` creates a `/var/treadmill/repo-overlays/<repo-slug>/` directory with a venv + `pip install` for `python`, `npm install --prefix` for `node`, and sha256-verified `urllib` downloads to `/var/treadmill/repo-bin/` for `binaries`. Cache key is a sha256 over the canonically-sorted `WorkerDeps`; an existing `.deps-hash` match short-circuits to `fresh=False` with the prior overlay paths. Per-stage failures wrap into `WorkerDepsMaterializationError(stage=...)`. The resolved `RepoOverlay` binds to a `_CURRENT_OVERLAY` ContextVar via `bind_overlay`; subprocess sites read it via `current_overlay()` and merge `env_overrides()` (`PATH` prepend + `PYTHONPATH` + `NODE_PATH`) onto the child env. ADR-0059 step 4: at the runner's materialize call-site (`runner._handle_step`), a `WorkerDepsMaterializationError` publishes `task.worker_deps_failed` (via `EventPublisher.publish_task_worker_deps_failed`) carrying `stage` + `detail` + `compute_deps_hash(worker_deps)`, then re-raises so `step.failed` still fires — distinct operator-visible signal vs gate-broken / architect_cap / stuck_task_sweep escalations. ADR-0060 step 3c: `_install_proxy_url()` reads the autoscaler-minted `TREADMILL_INSTALL_PROXY_TOKEN` plus the worker entrypoint's base `HTTPS_PROXY` and returns the credentialed URL `http://install:<token>@<host>:<port>`; every `materialize()` subprocess (`python -m venv`, `pip install`, `npm install`) is invoked with `env={**os.environ, "HTTPS_PROXY": <credentialed>, "HTTP_PROXY": <credentialed>}` and the binary `urllib.urlopen` is replaced with an opener built from `urllib.request.ProxyHandler({...credentialed...})`. The credential is the *only* signal the `treadmill-egress-proxy` honors when elevating to the install-phase allowlist (PyPI / npm / signed-binary hostnames); when the token is absent, `materialize()` leaves the subprocess env unchanged so the worker inherits the uncredentialed proxy (task-phase contract). The credential MUST NOT leak into any other subprocess seam — see `validation_runtime.run_deterministic`.
- `treadmill_agent/validation_runtime.py` — Deterministic + LLM-judge check execution. `run_llm_judge` injects the touched components' AGENT.md content into the judge prompt (via `gather_agent_md_context`) so rule prompts that reference an `AGENT_MD` input (e.g. ADR-0030's docs-current-with-pr) actually see the documentation they're asked to evaluate. `run_deterministic` merges `repo_deps.current_overlay().env_overrides()` onto the subprocess env (ADR-0059 step 2) so validation scripts pick up the repo's materialized venv / node_modules / binary set. ADR-0060 step 3c: `run_deterministic` deliberately does NOT touch `HTTPS_PROXY` / `HTTP_PROXY` — the validation subprocess inherits only the worker entrypoint's uncredentialed proxy, never the credentialed install URL `materialize()` builds. That asymmetry IS the task-phase contract: the egress proxy treats a request without the install credential as task-phase and applies the always-allowed allowlist only. Pinned by `tests/test_validation_runtime.py::test_deterministic_subprocess_env_excludes_install_credential`.
- `treadmill_agent/judge_eval.py` — Evaluation harness (ADR-0053) used by the agentic judge-prompt optimizer as its scoring metric. `evaluate_judge_prompt(prompt, examples, *, model, timeout_seconds)` runs a candidate prompt over labeled examples via `claude_code.run_claude`, parses each verdict from the JSON envelope (more permissive than `validation_runtime`'s parser — no fixed `Literal`), and case-insensitively matches it against each example's `gold_verdict`. Vocabulary-agnostic, so it scores both validator judges (`pass`/`fail`) and the architect (`accept-as-is`/`amend`). Parse failures and `run_claude` exceptions are surfaced as `error=True` in `per_example` and count against the score.

## Recent changes

- ADR-0061 v1.3 self-dispatching triage — `workers/agent/scripts/triage/plan-template-ui-fix.md` is a new bundled Plan template that ships into the agent image at `/opt/triage/plan-template-ui-fix.md` via the existing `COPY workers/agent/scripts/triage/ /opt/triage/` layer in `workers/agent/Dockerfile` (no Dockerfile edit). The role-ui-triage v1.3 prompt copies this file's body to `/tmp/triage-<run_id>/dispatched/<finding_seq>.md` per dispatched finding, fills its `<FINDING_ID_SHORT>` / `<SHORT_OBSERVATION>` / `<EVIDENCE_POINTER>` / `<PROPOSED_RESOLUTION>` / `<PROPOSED_RESOLUTION_FILES>` / `<COMPONENT_AGENT_MD>` / `<TARGET_PATH>` / `<PLAYWRIGHT_ASSERTION_DERIVED_FROM_PROPOSED_RESOLUTION>` placeholder seams from the finding being dispatched, then runs `treadmill plan validate` → `treadmill plan submit … --auto-merge=false` and threads the returned `plan_id` onto the finding's `dispatched_plan_id` before POSTing the findings batch. The template's single `sequence_of_work` entry targets `wf-author` (single-task, ui-fix shaped) and carries a deterministic Playwright validation block whose `script` launches `chromium`, navigates to `http://treadmill-dashboard:80/<TARGET_PATH>` (NOT `localhost:5174` — same container-DNS translation as the v1.2 prompt's network-mapping section), runs the placeholder-derived assertion, and exits non-zero on failure (severity=blocking, timeout 120 s). The literal `<...>` tokens are sentinels — the triage worker interpolates per finding; do not rename them without re-cutting the prompt artifact in lockstep. A submit failure downgrades the finding to `research_only` and the run continues; the at-most-3-dispatched cap from the prompt still applies.

Other roles don't need a Bash allowlist because their git ops route through `workers/agent/treadmill_agent/git.py` (Python, not Claude Bash) and their bulk file I/O goes through Edit/Write; triage's whole workflow IS Bash (probe + curl), which is why it's the first role to hit this wall. Patterns are narrow on purpose — widening is a deliberate ADR-style decision, not a slip.

## Pitfalls

- Each worker MUST exit after one step (EXIT_AFTER_STEP=true); persistent workers hide exit-path bugs that only surface in production autoscale churn. Never disable this in dev unless tracing a specific bug.
- Claude Code subprocess output parsing is fragile; changes to the Claude Code harness output format can break the worker silently (no exception, just lost events). Always test against a real Claude Code run after format changes.
- The workspace directory is created under `/tmp` and must be isolated per task to avoid concurrent writes during SQS-driven autoscaling. Cross-task workspace collisions cause data loss.
- Git credential injection via `startup_auth.py` happens once at container startup; if a step changes a secret or credential strategy mid-task, those changes will not be reflected in subsequent git operations within the same step.
