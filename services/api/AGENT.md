# services/api

## Purpose

This directory contains the Treadmill API, the event-driven control plane that coordinates workflow execution across the system. It is the single source of truth for workflow state, manages task dispatch, tracks step lifecycle, handles GitHub webhook ingestion, and routes work to agent workers. The API is built on FastAPI and SQLAlchemy async, backed by Postgres for durability and Redis for high-frequency state like task queues.

## Key surfaces

- `treadmill_api/cli.py` â€” entry point; starts the FastAPI server that listens for webhooks and exposes `/plans`, `/tasks`, `/health` routes.
- `treadmill_api/resolvers.py` â€” cross-entity resolvers; first resident `resolve_task_by_head_sha(session, repo, sha)` (task ec0e534c, the ADR-0063-deferred `(repo, head_sha)` lookup): task of the MOST-RECENT `task_prs` row matching `(repo, head_sha)`, else None. `task_prs.head_sha` is nullable (migration `20260612_0200`, partial index `ix_task_prs_repo_head_sha` over non-NULL) and only as fresh as its writers â€” the events-join fallback (pr_opened/pr_synchronize payloads) is deliberately left to consumers that need it. ADR-0090's CI-observer is the first consumer.
- `treadmill_api/events/` â€” event schema definitions (GitHub pushes, pull merges, plan documents, step lifecycle, task output). Event registry routes incoming webhooks to the appropriate consumer.
- `treadmill_api/events/schedule.py` â€” `ScheduledTick` payload (entity_type="schedule", action="tick") emitted by the scheduler on each cron fire (ADR-0035).
- `treadmill_api/scheduler/` â€” cron scheduler package (ADR-0035): `cron.py` (croniter wrapper), `policy.py` (deterministic jitter + quiet-hours + backoff, ported from RAMJAC), `runner.py` (`SchedulerRunner` â€” 30 s poll loop + 4 h missed-tick replay on startup; subprocess entrypoint now installs a rotating file handler + rate-limited error logger), `bounded_logging.py` (`configure_rotating_logging` + `RateLimitedErrorLogger`, mirroring the local-adapter helper but kept package-local so `treadmill_api` has no dependency on `treadmill_local`).
- `treadmill_api/dispatch.py` â€” task dispatch logic; decides when to enqueue a step, applies dependency gates, deduplicates by composite key (plan SHA + step name).
- `treadmill_api/models/run.py` â€” `WorkflowRun.task_id` is **nullable** for historical reasons (ADR-0035's original taskless path, now deprecated). Post-ADR-0057, both the scheduler and the operator-trigger endpoint create a **synthetic Task per dispatch** tied to `SYSTEM_PLAN_ID` and go through the normal `dispatcher.dispatch_task` path â€” every new `WorkflowRun` has a non-NULL `task_id`. The four historical orphan rows from before the fix are the only `task_id IS NULL` runs in the system. `WorkflowRunStep` also carries five nullable token-usage columns (ADR-0020) â€” `input_tokens` / `output_tokens` / `cache_creation_tokens` / `cache_read_tokens` (`BIGINT`) and `model` (`TEXT`) â€” populated by the coordination consumer from `step.completed.token_usage` and `NULL` for steps that made no LLM call.
- `treadmill_api/seed/system_plan.py` â€” ADR-0057: seeds the single "system" `Plan` (`SYSTEM_PLAN_ID = 00000000-0000-0000-0000-000000000001`) that parents every synthetic Task. `seed_system_plan_if_empty(session)` inserts both the `Plan` row and a `plan.activated` event in one txn â€” the activation lifts the system Plan into `derived_status='active'` so the `dispatch_task` plan-active gate (ADR-0010 / `dispatch.py`) doesn't park synthetic-task runs in deferred-dispatch. Auto-seeded at API startup between `_auto_seed_starters` and `_auto_seed_schedules` (`cli.py::_auto_seed_system_plan`).
- `treadmill_api/database.py` â€” Postgres schema: workflows, plans, steps, tasks, mergeability state, long-lived task output.
- `treadmill_api/parsers/plan_doc.py` â€” parses `docs/plans/*.md` frontmatter and extracts the task tree structure.
- `treadmill_api/onboarding_store.py` + `treadmill_api/models/onboarding.py` â€” ADR-0050 onboarding persistence: repo config (mode + auto-merge block), wf-discover repo profile, and the S3 context-doc index. Typed columns + Postgres ARRAYs per ADR-0011 (no JSONB). Also rounds-trips the ADR-0059 per-repo `worker_deps` (Python / Node TEXT[] arrays on `repo_configs` + `repo_worker_binaries` side table; binaries are replaced wholesale on every upsert; `get_repo_config` always materializes `worker_deps` as a non-None `WorkerDeps`). `RepoConfig` carries `claude_account` (ADR-0055) and `claude_account_fallback` (ADR-0066) â€” the fallback is the account the worker switches to when the primary hits a usage limit; `NULL` means no fallback configured.
- `treadmill_api/routers/onboarding.py` â€” ADR-0051 onboarding HTTP surface: `POST /api/v1/onboarding/repos` accepts a discovered profile + optional mode + optional `claude_account` (ADR-0055), upserts the repo profile and config via `OnboardingStore`, and resolves mode via `repo_profile.recommend_mode` when omitted. `GET /api/v1/onboarding/repos/{repo:path}` returns the persisted `RepoConfig` (mode, auto-merge block, commands, `claude_account`) or 404 â€” the mode-aware authoring skill (ADR-0054 d.4) calls it to route adapt vs conform; the worker's per-step credential fetch (ADR-0055) reads `claude_account` from the same surface.

- `treadmill_api/coordination/stuck_task_sweep.py` â€” `wf-stuck-task-sweep`, the first deterministic self-health bot (ADR-0035 P2, ADR-0047). - `treadmill_api/coordination/escalation_close_sweep.py` â€” `wf-escalation-close-sweep`, the ADR-0062 Step 2 close-detector. - `treadmill_api/coordination/terminal_gate_sweep.py` â€” `wf-terminal-gate-sweep`, the inverse-of-stuck-task self-health bot (ADR-0047, ADR-0038, ADR-0042). ## Pitfalls

- **Webhook ingress lock-step (ADR-0063):** Every new webhook ingress path â€” HTTP or SQS â€” MUST call `treadmill_api.webhooks.persist.persist_and_resolve_webhook_event` for the Event INSERT. Writing Event rows directly from a router or coordination module bypasses the FK-resolution + buffer-on-miss + publish seam and recreates the dual-ingress drift hazard that caused the 2026-05-29 Task 3b stall. The CI gate at `tests/test_webhook_ingress_lockstep.py` enforces this: it scans `routers/` and `coordination/` for direct writes and fails if any appear outside the curated allowlist of pre-existing, legitimate non-webhook writers (`coordination/consumer.py` and `coordination/triggers.py`). Do not extend the allowlist for new webhook paths.

- Event idempotency is load-bearing: the API can receive duplicate webhook deliveries from GitHub; the dispatch deduplication key must be stable across replays. Changes to dispatch dedup logic are high-risk.
- Mergeability state is recomputed on every step output; this recomputation touches dependent tasks and can trigger cascading redispatches. Bugs in the mergeability transition logic can create infinite loops.
- Alembic migrations run at container startup; if a migration hangs or fails, the API container will not start and the entire system will block. Always test migrations locally with a representative Postgres size.
- The plan document parser reads only the frontmatter; changes to the frontmatter schema or parsing logic can cause valid old plans to stop being recognized. Test against plans in the wild before merging.
