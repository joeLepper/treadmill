# services/api

## Purpose

This directory contains the Treadmill API, the event-driven control plane that coordinates workflow execution across the system. It is the single source of truth for workflow state, manages task dispatch, tracks step lifecycle, handles GitHub webhook ingestion, and routes work to agent workers. The API is built on FastAPI and SQLAlchemy async, backed by Postgres for durability and Redis for high-frequency state like task queues.

## Key surfaces

- `treadmill_api/cli.py` — entry point; starts the FastAPI server that listens for webhooks and exposes `/plans`, `/tasks`, `/health` routes.
- `treadmill_api/events/` — event schema definitions (GitHub pushes, pull merges, plan documents, step lifecycle, task output). Event registry routes incoming webhooks to the appropriate consumer.
- `treadmill_api/events/schedule.py` — `ScheduledTick` payload (entity_type="schedule", action="tick") emitted by the scheduler on each cron fire (ADR-0035).
- `treadmill_api/scheduler/` — cron scheduler package (ADR-0035): `cron.py` (croniter wrapper), `policy.py` (deterministic jitter + quiet-hours + backoff, ported from RAMJAC), `runner.py` (`SchedulerRunner` — 30 s poll loop + 4 h missed-tick replay on startup; subprocess entrypoint now installs a rotating file handler + rate-limited error logger), `bounded_logging.py` (`configure_rotating_logging` + `RateLimitedErrorLogger`, mirroring the local-adapter helper but kept package-local so `treadmill_api` has no dependency on `treadmill_local`).
- `treadmill_api/dispatch.py` — task dispatch logic; decides when to enqueue a step, applies dependency gates, deduplicates by composite key (plan SHA + step name).
- `treadmill_api/models/run.py` — `WorkflowRun.task_id` is **nullable** for historical reasons (ADR-0035's original taskless path, now deprecated). Post-ADR-0057, both the scheduler and the operator-trigger endpoint create a **synthetic Task per dispatch** tied to `SYSTEM_PLAN_ID` and go through the normal `dispatcher.dispatch_task` path — every new `WorkflowRun` has a non-NULL `task_id`. The four historical orphan rows from before the fix are the only `task_id IS NULL` runs in the system. `WorkflowRunStep` also carries five nullable token-usage columns (ADR-0020) — `input_tokens` / `output_tokens` / `cache_creation_tokens` / `cache_read_tokens` (`BIGINT`) and `model` (`TEXT`) — populated by the coordination consumer from `step.completed.token_usage` and `NULL` for steps that made no LLM call.
- `treadmill_api/seed/system_plan.py` — ADR-0057: seeds the single "system" `Plan` (`SYSTEM_PLAN_ID = 00000000-0000-0000-0000-000000000001`) that parents every synthetic Task. `seed_system_plan_if_empty(session)` inserts both the `Plan` row and a `plan.activated` event in one txn — the activation lifts the system Plan into `derived_status='active'` so the `dispatch_task` plan-active gate (ADR-0010 / `dispatch.py`) doesn't park synthetic-task runs in deferred-dispatch. Auto-seeded at API startup between `_auto_seed_starters` and `_auto_seed_schedules` (`cli.py::_auto_seed_system_plan`).
- `treadmill_api/database.py` — Postgres schema: workflows, plans, steps, tasks, mergeability state, long-lived task output.
- `treadmill_api/parsers/plan_doc.py` — parses `docs/plans/*.md` frontmatter and extracts the task tree structure.
- `treadmill_api/onboarding_store.py` + `treadmill_api/models/onboarding.py` — ADR-0050 onboarding persistence: repo config (mode + auto-merge block), wf-discover repo profile, and the S3 context-doc index. Typed columns + Postgres ARRAYs per ADR-0011 (no JSONB). Also rounds-trips the ADR-0059 per-repo `worker_deps` (Python / Node TEXT[] arrays on `repo_configs` + `repo_worker_binaries` side table; binaries are replaced wholesale on every upsert; `get_repo_config` always materializes `worker_deps` as a non-None `WorkerDeps`). `RepoConfig` carries `claude_account` (ADR-0055) and `claude_account_fallback` (ADR-0066) — the fallback is the account the worker switches to when the primary hits a usage limit; `NULL` means no fallback configured.
- `treadmill_api/routers/onboarding.py` — ADR-0051 onboarding HTTP surface: `POST /api/v1/onboarding/repos` accepts a discovered profile + optional mode + optional `claude_account` (ADR-0055), upserts the repo profile and config via `OnboardingStore`, and resolves mode via `repo_profile.recommend_mode` when omitted. `GET /api/v1/onboarding/repos/{repo:path}` returns the persisted `RepoConfig` (mode, auto-merge block, commands, `claude_account`) or 404 — the mode-aware authoring skill (ADR-0054 d.4) calls it to route adapt vs conform; the worker's per-step credential fetch (ADR-0055) reads `claude_account` from the same surface.
- `treadmill_api/routers/context_docs.py` — ADR-0054 context-doc REST API: `PUT/GET/LIST /api/v1/repos/{repo}/docs[/{doc_path:path}]` for per-repo context docs. PUT writes content to S3 via `ContextStore` and inserts the next version row via `OnboardingStore.record_context_doc`. GET returns a presigned-URL pointer + version (404 when the row is missing). LIST returns the latest version per `doc_path` for the repo. All three return 503 when `CONTEXT_DOCS_BUCKET` is unset (mirrors the GitHub App "service not configured" pattern in `routers/github.py`).
- `treadmill_api/routers/claude_credentials.py` — ADR-0055 Claude credential resolver: `POST /api/v1/claude/credentials {repo}` → `{account, type, token, fallback?}`. Worker calls this at its per-step re-mint seam to pick the right account's credential (OAuth long-lived token or Anthropic API key) and inject `CLAUDE_CODE_OAUTH_TOKEN` / `ANTHROPIC_API_KEY` into the Claude Code subprocess env. Config comes from `Settings.claude_accounts_json` + `claude_default_account`; secrets are fetched from Secrets Manager. Failure modes are explicit (503 unconfigured, 404 unknown account, 502 SM error) — never a silent cross-account fallback on the primary path. Optional `fallback` block (ADR-0066): when `RepoConfig.claude_account_fallback` is set, the response also carries `{account, type, token}` for the fallback credential resolved best-effort — any failure (account not in map, SM error, empty SecretString) logs a warning and leaves `fallback: null`; it never turns a working primary into an error.
- `treadmill_api/coordination/triggers.py` — auto-merge trigger (ADR-0031) now also honors the ADR-0050 per-repo `auto_merge_blocked` config in addition to the plan-level flag. Both the deadline-arming path (`maybe_auto_merge_on_mergeable`) and the fire-time gate (`_check_still_mergeable_for_auto_merge`) consult `OnboardingStore.get_repo_config` via the `_repo_auto_merge_blocked` helper; missing config or any lookup error fails OPEN to preserve pre-ADR-0050 behavior. ADR-0062 Step 1 adds `maybe_dispatch_terminal_step_failure_escalation` — a `step.failed`-keyed producer that emits `task.escalated_to_operator` with `reason='terminal_step_failure'` + `step_name` + the failing step's captured error as `gate_log_excerpt` when the owning run has no remaining pending steps AND no sibling escalation event has fired against the task within a 5-minute dedup window (the cap-reached / gate-broken producers own their cases — this covers the leftover "run ran out of recovery and nobody else raised" path). Consumer wires it as a sibling of `_maybe_fire_author_feedback_on_step_failed` in the `step.failed` branch.
- `treadmill_api/coordination/stuck_task_sweep.py` — `wf-stuck-task-sweep`, the first deterministic self-health bot (ADR-0035 P2, ADR-0047). Driven by the seeded `*/10` schedule; `handle_scheduled_tick` short-circuits on the `wf-stuck-task-sweep` slug to `run_stuck_task_sweep` instead of looking up a `WorkflowVersion` (no role step — the silent-stall signal is a pure query). Detects non-terminal tasks whose latest `step` event is `step.completed` with no later `step.ready` and whose last activity is older than `STUCK_TASK_THRESHOLD` (30 min); emits one `task.escalated_to_operator` per stall via `_emit_operator_escalation`. Idempotent: the SQL excludes any task already carrying an `escalated_to_operator` event.
- `treadmill_api/coordination/escalation_close_sweep.py` — `wf-escalation-close-sweep`, the ADR-0062 Step 2 close-detector. Driven by the seeded `*/2` schedule (close latency matters more than throughput); `handle_scheduled_tick` short-circuits on the slug to `run_escalation_close_sweep` (same idiom as the stuck-task sweep — pure query, no role step). For each open incident (latest `task.escalated_to_operator` without a later `task.escalation_closed` for the same task) checks the four sweep-detectable close triggers in priority order — `re_progressed` (a `step.completed` after `opened_at`) → `pr_merged` (a `github.pr_merged` for the task) → `cancelled` (a `task.cancelled`) → `superseded` (a `task.superseded`) — and emits one `task.escalation_closed` per match with `mttr_seconds = (now - opened_at).total_seconds()` stamped on the payload. The fifth trigger, `operator_close`, is emitted by the Step-3 CLI via the shared `emit_operator_close` helper (single-sourced so MTTR computation lives in one place). Idempotent at the SQL layer: incidents that already have a later close event are excluded from the open-incidents SELECT.
- `treadmill_api/coordination/terminal_gate_sweep.py` — `wf-terminal-gate-sweep`, the inverse-of-stuck-task self-health bot (ADR-0047, ADR-0038, ADR-0042). Driven by the seeded `*/10` schedule; `handle_scheduled_tick` short-circuits on the `wf-terminal-gate-sweep` slug to `run_terminal_gate_sweep` (same idiom as the other deterministic sweeps — pure query, no role step). Detects tasks that received an architect accept-as-is verdict (`review.override` per ADR-0038 or `validate.override` per ADR-0042) but whose PR was never merged (`github.pr_merged` absent), excluding `cancelled` and `superseded` tasks (those legitimately leave a PR unmerged). Emits one `task.escalated_to_operator` with `reason='terminal_gate_sweep'` per orphaned PR via `_emit_operator_escalation`. Idempotent: the SQL `NOT EXISTS` dedup is signal-specific (`payload->>'last_verdict' = :signal`) so prior stuck-task escalations on the same task don't mask a new orphan escalation. The `escalation_close_sweep` auto-closes the incident once `github.pr_merged` arrives via the `pr_merged` close trigger. `TERMINAL_GATE_SIGNAL = "wf-terminal-gate-sweep-orphaned-pr"`.
- `treadmill_api/coordination/notification_fanout.py` — ADR-0062 Step 4 Slack + pluggable-webhook fan-out for the escalation incident lifecycle, extended by ADR-0071 with a sibling Telegram target. `NotificationFanout` is a background subscriber over the in-process eventbus broadcaster (`subscribe_local` / `unsubscribe_local` in `eventbus.py`, the same seam the dashboard WS uses, ADR-0056) so the API doesn't need a fresh SQS queue + subscription filter to push escalation signals to operator channels. Reacts to `task.escalated_to_operator` and `task.escalation_closed` only; every other event published in-process is dropped at the gate (the ADR-0071 server fan-out is escalations-only; non-escalation events do NOT relay from the server — the per-session relay covers richer detail). On a match: if `TREADMILL_SLACK_WEBHOOK_URL` is set, POST a Slack-formatted one-liner JSON (`:rotating_light:` on open / `:white_check_mark:` on close, 8-char task-id snippet, `reason` from payload, `mttr=<n>s` on close). For each URL in `TREADMILL_NOTIFICATION_WEBHOOKS` (comma-separated, may be empty), POST the raw `_build_record`-shaped event JSON so downstream consumers (PagerDuty bridges, custom routers) see the same dict SNS would have carried. If BOTH `TREADMILL_TELEGRAM_BOT_TOKEN` and `TREADMILL_TELEGRAM_CHAT_ID` are set (ADR-0071), POST `{chat_id, text}` to `https://api.telegram.org/bot<token>/sendMessage` where `text` is the same one-liner as the Slack body minus Slack's `:emoji:` shortcodes (plain words); either Telegram field unset = no Telegram hop. The bot token is a secret: read from settings, never logged (start-time log records only "configured"/"unconfigured"; error logs include only event_id + action). Per-target failures log + continue — a wedged Slack endpoint never blocks the generic-webhook fan-out, one bad URL never poisons the others, a Telegram failure never blocks Slack/raw and vice versa, and the subscriber never raises into the broadcaster. Owned `httpx.AsyncClient` (5 s per-POST timeout) built in `start()` and closed in `stop()`; injected clients (tests / future shared client) are left alone. `start()` is a no-op when neither Slack, generic webhooks, nor Telegram is configured so a deployment without targets pays nothing. Wired into the FastAPI lifespan in `app.py` alongside the existing publisher / consumer / replay-loop / webhook-inbox-poller — order: started after the inbox poller, stopped before `github_clients.aclose()`. Env-var contract: `TREADMILL_SLACK_WEBHOOK_URL` (str, optional; empty/unset → no Slack hop), `TREADMILL_NOTIFICATION_WEBHOOKS` (comma-separated str, default `""`; parsed via `Settings.notification_webhook_urls`), `TREADMILL_TELEGRAM_BOT_TOKEN` + `TREADMILL_TELEGRAM_CHAT_ID` (both optional; only the pair-set case enables the Telegram hop).
- `treadmill_api/webhooks/pending_events.py` — cache-then-heal Redis-list buffer per ADR-0007 / ADR-0063. After ADR-0063 Step 2 the public surface is **opaque-key**: `buffer_pending_event(redis, pending_buffer_key, event_id)`, `drain_pending_events(redis, session, publisher, pending_buffer_key, task_id)`, and `pending_event_count(redis, pending_buffer_key)` all take an arbitrary `str` key — the module no longer hard-codes the (repo, pr_number) shape. The original PR-bound shape lives on as the helper `pr_pending_buffer_key(repo, pr_number) -> "pr:{repo-lower}:{pr_number}:pending_events"` and is what all four current callers (`routers/webhooks.py`, `coordination/webhook_inbox.py`, `coordination/consumer.py` × 2) thread through to the opaque-key API. Future buffers keyed on plan SHA / issue id / etc. can reuse the same Redis-list mechanics without forking the module. TTL is `PENDING_TTL_SECONDS = 48h` (matches bunkhouse).

- `treadmill_api/webhooks/persist.py` — **ADR-0063 Step 3 shared seam** for webhook ingress. `persist_and_resolve_webhook_event(session, normalized, body_json, redis_client, publisher, *, event_id=None) -> Event` performs the case-insensitive `task_prs(repo, pr_number)` lookup, persists the `events` row with the resolved `task_id` (or NULL), buffers the event_id on the pending list when the lookup misses (cache-then-heal per ADR-0007), and publishes on the bus (publish failure → log + swallow; the row is the source of truth, the consumer's rescan picks it up). Optional `event_id` kwarg switches between the HTTP-path fresh-UUID `session.add(Event(...))` and the SQS-path deterministic-id `pg_insert(...).on_conflict_do_nothing(index_elements=["id"])` — same helper, both ingress paths. Both `routers/webhooks.py` and `coordination/webhook_inbox.py::WebhookInboxPoller._persist_and_publish` delegate to this function; the lock-step requirement that closed the 2026-05-29 Task 3b dual-ingress drift is now structural rather than maintained by code review.
- `treadmill_api/starters.py` — `role-prompt-optimizer` + `wf-tune-judge-prompts` (ADR-0053 Wave 2). Single-step workflow that proposes ONE refined variant of a target judge's prompt, scores both against the held-out slice of the labeled corpus via `evaluate_judge_prompt` (in `workers/agent/treadmill_agent/judge_eval.py`), and either opens a PR with the rule-YAML patch (when the variant beats current by ≥0.05) or reports `no_improvement`. Operator-triggered today with payload `{judge_role, corpus_s3_uri}`; no schedule yet (Wave 3). The role is sonnet-tier (structured-output reliability + rarely dispatched).
- `treadmill_api/routers/workflow_triggers.py` — ADR-0053 Wave 3 + ADR-0057 operator surface: `POST /api/v1/workflows/{slug}/trigger` body `{payload: dict}` enforces `payload.repo` is non-empty (400 otherwise) and calls `coordination/triggers.py::_dispatch_via_synthetic_task` with `created_by="operator-trigger"`. The helper creates a synthetic `Task` under `SYSTEM_PLAN_ID`, emits `task.registered`, and goes through the normal `dispatcher.dispatch_task` path so the operator surface and the scheduled-bot surface share one task-bound dispatch path (workers always see a non-NULL `task_id`). Returns `{run_id, task_id, workflow_id}` 201; the `task_id` lets operators grep events / audit log for the dispatch they just kicked off. CLI wrapper: `treadmill workflows trigger <slug> --payload '<json>'`.
- `treadmill_api/routers/dashboard/` — ADR-0056 operator dashboard endpoints. The package's `__init__.py` exposes ONE aggregator `router = APIRouter(prefix="/api/v1/dashboard", tags=["dashboard"])` and uses `pkgutil.iter_modules` + `importlib.import_module` to auto-discover every sibling `.py` module that exports a top-level `router = APIRouter()` — drop a file in and it gets mounted with no `__init__.py` or `app.py` edit (the contract PR-B2..B5 rely on for zero-shared-edit fan-out). `routers/dashboard/overview.py` is the first endpoint: `GET /api/v1/dashboard/overview` returns `{accounts, fleet, escalations, tasks, bucketCounts, events}` matching `services/dashboard/src/api/types.ts` `useOverview` field-for-field, honoring optional `repo` / `bucket` (`blocked|inflight|hopper`) / `account` / `q` / `reason` (`architect_cap|stuck_task_sweep|gate-broken`, ADR-0058 Step 5 — narrows the `escalations` array only) query params; operator-bucket derivation mirrors `mock.ts` `operatorBucket()` exactly. Fleet heartbeats + per-account spend are best-effort/stubbed pending the persistence work to back them.
- `treadmill_api/routers/escalations.py` — ADR-0062 Step 3 CLI-shaped escalation surface mounted at `/api/v1/escalations` (operator dashboard remains at `/dashboard/overview`; this router backs the `treadmill escalations` CLI group, not the dashboard tile). Five endpoints: `GET /` lists open incidents (mirrors the dashboard's `_ESCALATIONS_SQL` plus the close-event filter — an incident is open iff its most recent `task.escalated_to_operator` has no later `task.escalation_acknowledged` AND no later `task.escalation_closed`); optional `?reason=` + `?task=` (case-insensitive prefix) narrows in Python so the single SQL hot path stays shared. `GET /stream` is the SSE feed for `treadmill escalations tail` — subscribes to `eventbus.subscribe_local` (same in-process publisher seam as `routers/dashboard/ws.py`), filters to `escalated_to_operator` / `escalation_acknowledged` / `escalation_closed` actions, emits one `data:`-prefixed JSON frame per matching event and a `: ping` comment every `heartbeat_interval` seconds; never replays history (pair with `GET /` for the bootstrap snapshot). `POST /{task_id}/close` routes through the shared `emit_operator_close` helper from `coordination/escalation_close_sweep.py` (single-sourced MTTR + emitter so the CLI close and the sweep stay byte-identical); returns 404 missing task, 409 no open incident, 409 already-closed. `POST /{task_id}/ack` emits `task.escalation_acknowledged` via `dispatcher.persist_and_publish` so the SSE stream observes the ack (the dashboard's ack endpoint at `routers/dashboard/ack_escalation.py` writes via raw INSERT instead — both paths converge on the same overview SQL filter). `GET /report` aggregates every `task.escalation_closed` event since `?since=` (defaults to 7 days ago UTC) and buckets by `?by=reason|day|task`, returning `count` + `mttr_seconds_avg/p50/p95` per bucket with nearest-rank percentiles computed in Python (single sorted list per bucket; Postgres `percentile_disc` would work too but bloats the SQL for no operator-scale gain). The `/stream` endpoint is the only surface in this router that lives off the publisher rather than the events table — every other endpoint is a pure read of the events log.

- Terminal-gate orphan sweep (ADR-0047, ADR-0038, ADR-0042): new `coordination/terminal_gate_sweep.py` — `wf-terminal-gate-sweep`, the third deterministic self-health bot. Driven by the seeded `*/10` schedule; `handle_scheduled_tick` gains a third short-circuit branch (after the stuck-task and escalation-close sweeps) for the `wf-terminal-gate-sweep` slug. ## Pitfalls

- **Webhook ingress lock-step (ADR-0063):** Every new webhook ingress path — HTTP or SQS — MUST call `treadmill_api.webhooks.persist.persist_and_resolve_webhook_event` for the Event INSERT. Writing Event rows directly from a router or coordination module bypasses the FK-resolution + buffer-on-miss + publish seam and recreates the dual-ingress drift hazard that caused the 2026-05-29 Task 3b stall. The CI gate at `tests/test_webhook_ingress_lockstep.py` enforces this: it scans `routers/` and `coordination/` for direct writes and fails if any appear outside the curated allowlist of pre-existing, legitimate non-webhook writers (`coordination/consumer.py` and `coordination/triggers.py`). Do not extend the allowlist for new webhook paths.

- Event idempotency is load-bearing: the API can receive duplicate webhook deliveries from GitHub; the dispatch deduplication key must be stable across replays. Changes to dispatch dedup logic are high-risk.
- Mergeability state is recomputed on every step output; this recomputation touches dependent tasks and can trigger cascading redispatches. Bugs in the mergeability transition logic can create infinite loops.
- Alembic migrations run at container startup; if a migration hangs or fails, the API container will not start and the entire system will block. Always test migrations locally with a representative Postgres size.
- The plan document parser reads only the frontmatter; changes to the frontmatter schema or parsing logic can cause valid old plans to stop being recognized. Test against plans in the wild before merging.
