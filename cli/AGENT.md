# cli

## Purpose

The Treadmill CLI (`treadmill`) is the operator's interface to the Treadmill API (per ADR-0010). It wraps HTTP calls to the API and presents results via Rich tables for human consumption.

## Key surfaces

- `treadmill_cli/cli.py` — main entrypoint; registers all command groups and defines plan/task/workflow/role subcommands inline.
- `treadmill_cli/api_client.py` — thin `ApiClient` wrapper over `httpx`; all API calls go through `_request`. Covers plans, tasks, workflows, onboarding repos, and health endpoints.
- `treadmill_cli/commands/learnings.py` — `treadmill learnings crystallize` (ADR-0034).
- `treadmill_cli/commands/schedules.py` — `treadmill schedules list|create|pause|resume|delete` (ADR-0035).
- `treadmill_cli/commands/onboarding.py` — `treadmill onboarding update <repo>` for managing per-repo worker deps (ADR-0059 Step 5): `--worker-deps-python`, `--worker-deps-node`, `--binary name=URL=SHA256@TARGET`, `--clear-worker-deps`.
- `treadmill_cli/commands/team.py` — `treadmill team up <owner/name> [--workers N] [--force]` (ADR-0087 §Team bootstrap, PR-B). Provisions a per-repo Treadmill team: one coordinator + one evaluator + N workers (default 3), each with its own systemd unit + per-session directory tree. Steps: (1) derive `slug` = `repo` with `/` replaced by `-` and lowercased; deterministically derive `coordinator-<slug>`, `evaluator-<slug>`, `worker-<slug>-1` … `worker-<slug>-N` (no manual override — ADR-0087 mandates deterministic label families); (2) `POST /api/v1/team_configs` upserts the row with all four label families (the router enforces the scale-down guard server-side, returning 409 if shrinking `worker_labels` would orphan in-flight `task_executions` rows; `--force` forwards as `?force=true`); (3) per session label, create `~/.treadmill/teams/<slug>/<label>/` with two files: `.session-id` (empty stub on first creation; preserved across re-runs once the coordinator has captured the real Claude Code session ID — overwriting would lose worker memory) and `<label>.env` (always rewritten with `TREADMILL_ROLE` derived from label prefix, `TREADMILL_LABEL`, `TREADMILL_API_URL`); (4) `systemctl --user enable/start treadmill-channel@<label>.service` for each session. `TREADMILL_API_URL` is read from env (fallback `http://localhost:8000`). Idempotent: the upsert preserves the row, `mkdir(parents=True, exist_ok=True)` for the tree, `.session-id` stubs are not overwritten, env files rewrite, systemctl no-ops on already-enabled/running units. Scale-up just adds the new `worker-<slug>-K` labels; scale-down is gated by the server-side guard (operator must `--force` to override). Systemd unavailability (FileNotFoundError, non-zero rc) warns to stderr but does not raise — the load-bearing artifacts (API row + directory tree) survive and the systemd hop is retryable by hand. Tests in `cli/tests/test_cli_team_up.py` (15 tests) cover label derivation (slug case + slash-replacement; default 3 workers; custom count; role inference from label prefix), happy path (POST body has all four label families; directory tree per label with both files; env files carry correct role; systemctl enable+start fires per session — 1 coord + 1 eval + N workers × 2 verbs each), idempotency (`.session-id` populated by the coordinator survives a re-run), scale-down (CLI exits 2 on 409; `--force` appends the query param), input validation (no slash → exit 1; `--workers 0` rejected by Typer's `min=1`), systemd unavailable (warns but succeeds; directory tree still populated).
- `treadmill_cli/commands/repo.py` — DEPRECATED alias for `treadmill team up`. `treadmill repo add <owner/name> [--workers N] [--force]` emits a `DeprecationWarning` and delegates verbatim to `team.up`. Preserved as a compatibility shim during the ADR-0087 deprecation window; removed in the next CLI minor version (ADR-0087 Plan Phase 5). Tests in `cli/tests/test_cli_repo_add.py` (2 tests) pin the alias contract — warning fires; underlying team-up POST body has the ADR-0087 derived-labels shape; `--workers` threads through.
- `treadmill_cli/migration_chain.py` + `treadmill plan check-migration-chain --versions-dir <path>` — Alembic migration-chain linter (ADR-0085+0086 plan post-mortem surprise C). Parses every migration file in the versions directory, builds the chain graph, and reports three categories of structural problem: **multi-head** (chain has >1 terminal revision — the exact failure mode that bit Bert + Carla on 2026-06-09); **duplicate-revision** (two files declare the same `revision` identifier); **dangling-down-revision** (a migration's `down_revision` references an id no file declares). The multi-head check counts terminal heads (revisions no other file claims as parent), NOT shared parents — alembic's merge-migration shape (`down_revision = ("a", "b")` tuple, the output of `alembic merge`) is a legitimate reconciliation of a multi-head condition and the linter must not false-positive on it. The down_revision parser accepts all three concrete forms: `None` (root), `"abc"` (linear), and `("a", "b")` / `["a", "b"]` (merge). Pure-files check — no DB, no Alembic import — so it runs in the worker sandbox, pre-commit hook, or CI step without setup. Regex accepts both typed (`revision: str = "..."`) and untyped (`revision = "..."`) annotation styles; ignores `__init__.py` and non-Python files. Public API: `find_chain_violations(versions_dir: Path) -> list[ChainViolation]` where each violation has `kind`, `revisions: tuple[str, ...]`, `files: tuple[Path, ...]`, `detail`. CLI exit codes: 0 clean, 1 violations, 2 directory missing. Tests in `cli/tests/test_migration_chain.py` cover all three violation kinds + clean cases + merge-migration reconvergence (the original false-positive case) + annotation-style robustness + CLI wrapper exit codes (18 unit tests, `tmp_path` fixtures, no real DB).
- `treadmill_cli/commands/escalations.py` — `treadmill escalations [list|tail|close|ack|report]` (ADR-0062 Step 3) wraps the `/api/v1/escalations` surface. `list` snapshots open incidents (optional `--reason`, `--task` prefix). `tail` does snapshot-then-SSE-delta: first GETs `/api/v1/escalations`, then opens `/api/v1/escalations/stream` and prints colored one-liners per `escalated_to_operator` / `escalation_acknowledged` / `escalation_closed` event. `close` posts `/api/v1/escalations/{task_id}/close` (server emits `escalation_closed` with `close_reason='operator_close'` + computed MTTR). `ack` posts `/api/v1/escalations/{task_id}/ack`. `report --by reason|day|task --since ISO` renders the MTTR aggregation as a Rich table with `count`, `MTTR avg/p50/p95` columns formatted as `HhMMm` / `MMmSSs` / `SSs` per the magnitude of the bucket.
- `treadmill_cli/commands/tokens.py` — `treadmill tokens harvest|report` (ADR-0089 §2 token meter). `harvest [--projects-dir ~/.claude/projects]` walks session transcript JSONL beyond each file's API-remembered byte cursor, extracts per-call usage from assistant-message `usage` fields (deduping the multiple streamed lines that share one `requestId` — last line wins), attributes the session label from the project-dir name (rightmost `worker-`/`coordinator-`/`evaluator-`/`orchestrator-` prefixed segment; non-team dirs keep the raw dir name), joins each call to a `task_executions` window (label + `started_at`..`completed_at` containing `called_at`) when **exactly one** matches — zero/ambiguous leave `task_execution_id` NULL — and POSTs one batch per file to `/api/v1/llm_calls/harvest`. Unparseable lines are counted per run and accumulated server-side, never silently skipped (ADR-0089). The byte cursor only advances past complete (newline-terminated) lines; a straddle re-send hits the API's `(transcript_path, request_id)` unique index and updates the earlier (undercounted) row's usage in place. The projects dir is canonicalized (`resolve()`) so a path-spelling change can't fork the cursor/index keys; the malformed count is sent as the cumulative per-file value (server overwrites — retry-idempotent). `report --since <ISO date/datetime>` renders the per-label rollup (calls, output, fresh-in, cache-creation, cache-read, hit-ratio) plus the malformed-line total.
- `treadmill_cli/observe.py` — `treadmill observe` read-only Grafana access layer (ADR-0020).
- `treadmill_cli/config.py` — config loading from env vars (`TREADMILL_API_URL`, `TREADMILL_API_KEY`).

## Recent changes

- **ADR-0089 §2 — `treadmill tokens` group (harvester + standing report)**: new `commands/tokens.py` (see Key surfaces) + `api_client.py` methods `list_harvest_cursors` / `harvest_llm_calls` / `token_report` / `list_task_executions(worker_label=)`. Idempotency is two-layer: the `llm_harvest_cursors` byte offset (primary — re-runs parse only appended bytes; a cursor at EOF skips the file with no POST) and the server's `ON CONFLICT DO UPDATE` on `(transcript_path, request_id)` (last-write-wins backstop for responses still streaming when a run snapshots the file — the re-send corrects the mid-stream undercount). Paths are canonicalized via `resolve()`; the malformed-line count travels as a cumulative absolute value the server overwrites. The window-join is deliberately exactly-one: orchestrator/coordinator calls have no execution row and stay NULL (the nullable-FK decision is documented in `services/api` migration `20260611_0600` + `models/llm_call.py`). Tests `tests/test_tokens_command.py` (17, fixture JSONL + `pytest_httpx`): label attribution (team dirs, raw fallback, worktree `workers-agent` non-match), happy path (requestId dedupe, cursor offset = file size), idempotent rerun posts nothing, cursor resume mid-file, malformed lines counted + surfaced, partial trailing line left for next run, window-join unit + end-to-end (in-window joined / out-of-window NULL / ambiguous NULL / open window joined), report rendering + `--since` validation. Console output assertions are width-insensitive (wide-console fixture + whitespace-normalized `_plain`).

- **REMOVED 2026-06-11 (ADR-0088 unwind)** — `treadmill promote` deleted; deploy approval is GitHub environment protection. Original entry: **ADR-0088 `treadmill promote` group** — `commands/promote.py`: `list`/`show`/`approve`/`reject` against `/api/v1/prod_promotions`. The CLI is the single write path for promotion decisions: approve/reject send `X-Operator-Key` from `TREADMILL_OPERATOR_KEY` (refused client-side with no API call when unset). `approve` records the decision then fires `gh workflow run promote-to-prod.yml -f proposal_id=...` — order matters; a failed dispatch reports approval-IS-recorded + safe re-run (the endpoint is idempotent, the workflow's CAS prevents double-deploys). `--no-dispatch` records only. `show` renders the bundle with the diff_summary front and center — that list is what the operator is actually approving. Tests: `tests/test_promote_command.py` (8 — key header, order, dispatch-failure messaging, no-dispatch, 409 surface).

- **ADR-referenced API surface gap check (post-mortem surprise B)**: new `cli/treadmill_cli/adr_api_coverage.py` heuristic that audits ADR-referenced HTTP endpoints against the live router inventory. Triggered automatically by `treadmill plan validate` when a plan doc references any `ADR-XXXX` ID — extracts `POST/GET/PATCH/DELETE/PUT /api/v1/...` references from the cited ADR, walks `services/api/treadmill_api/routers/*.py` combining each `APIRouter(prefix=...)` with its `@router.<verb>(...)` decorators, and reports unmatched references as WARNINGS (not violations — ADRs can legitimately reference future endpoints as design intent). Path normalisation collapses `{repo:path}`→`{repo}`, `{id}`/`{step_id}` parameter-name shorthand to a positional wildcard, and trailing-slash variants, so the false-positive class from cosmetic differences is eliminated. Standalone CI helper at `tools/check-adr-api-coverage.py` runs the same check ad-hoc — exit 0 clean, 1 on gaps, 2 on bad usage. 29 unit tests at `cli/tests/test_adr_api_coverage.py` cover ADR ref extraction, endpoint regex, router-inventory walk (prefix + suffix combination, dunder-file skip, multi-file aggregation), diff with all three normalisation cases (including the positional-gap-preserved counter-test), and the top-level orchestrator's canonical-gap / no-refs-no-op / clean / multi-ADR / missing-dirs paths. Dogfooded against the merged ADR-0085+0086 combined plan — correctly catches `POST /api/v1/events` (ADR-0086 §12.4 Path B backstop, intentionally deferred) without false-positiving on the `{id}` vs `{step_id}` workflow_run_steps shorthand. Task H of the post-mortem cleanup.

- **ADR-0085+0086 Task D — `treadmill plan submit` participates in auto-routing without flag changes**: no CLI surface changes were required for Task D. The CLI's existing `plan submit` path POSTs to `/api/v1/plans` which now reads `team_configs` server-side: if the repo has a coordinator registered (via `treadmill repo add ...`, Task F), the request body's omitted `created_by` is auto-derived from `team_config.coordinator_label` on the server, and a `plan.submitted` event is published for the coordinator. CLI operators can still pass `--created-by <label>` to override, but the standard flow no longer requires it when the team is registered. See `services/api/AGENT.md` Task D entry for the server-side wire.

- **ADR-0081 §1 operator_note CLI surface**: new `treadmill task note <task_id> "text"` subcommand sets the operator note on a task; `--clear` flag clears it (sets to null). Shares the existing `task_app` group and uses `ApiClient.set_operator_note` wrapper around `POST /api/v1/tasks/{task_id}/operator_note`. `treadmill onboarding update` gains `--hints on|off` flag to flip `worker_hints_enabled` per-repo (persisted via PATCH `/api/v1/onboarding/repos/{repo}`). Tests: `tests/test_cli_task_note.py` covers set, clear, 404 on missing task, and error surfaces; `tests/test_cli_onboarding_update_hints.py` covers hints flag routing to API via the ApiClient layer.

- ADR-0068 identity scheme — new `treadmill_cli/identity.py` module with `resolve_created_by(explicit)`. Per ADR-0068, the session label (= bot name = `created_by`) must be consistent for the channel-server WS filter to match. Inside a labeled session (`TREADMILL_SESSION_LABEL=<name>`), omit `--created-by` and the right tag is automatic; passing a mismatching `--created-by` emits a loud stderr warning. Both `plan submit` (`cli.py`) and `schedules create` (`commands/schedules.py`) now call `resolve_created_by` instead of using the raw typer option or the inline `os.environ.get("USER") or "operator"` fallback. Tests in `tests/test_identity.py` (6 unit tests covering all 5 resolution rules) and `tests/test_cli_plan_submit.py` (integration test asserting API body carries the session label).

- ADR-0062 Step 3 (escalations CLI group): new `commands/escalations.py` adds `treadmill escalations [list|tail|close|ack|report]` over the new `/api/v1/escalations` router. `list` snapshots open incidents with optional `--reason` (`architect_cap` / `stuck_task_sweep` / `gate-broken`) + `--task` prefix filters; `tail` does snapshot-then-SSE-delta against `/api/v1/escalations/stream`, opening a streaming `httpx.Client` directly (the shared `ApiClient` is a blocking single-shot client and is the wrong tool for a long-lived SSE socket); `close` and `ack` POST to the per-task action endpoints; `report` renders the MTTR aggregation (`avg / p50 / p95`) as a Rich table with `_fmt_mttr` formatting (`SSs` / `MMmSSs` / `HhMMm` magnitudes). CLI registered in `cli.py` next to `schedules_app` / `onboarding_app`. Tests in `tests/test_escalations_command.py` cover list (empty / table / `--reason` / `--task` / API error), close (happy / 404 / 409), ack (happy / 409), report (by reason / empty / `--by day` param wiring / `--since` round-trip / invalid `--since` / invalid `--by`), and tail (snapshot + one streamed SSE frame / empty snapshot + empty stream / 503 stream-connect failure surfaces "stream connect failed").
- [#65](https://github.com/joeLepper/treadmill/pull/65) — ADR-0059 Step 5 (operator CLI for worker deps): new `commands/onboarding.py` adds `treadmill onboarding update <repo>` with `--worker-deps-python` / `--worker-deps-node` / `--binary name=URL=SHA256@TARGET` (repeatable, additive, deduplicated) and `--clear-worker-deps` (mutually exclusive with dep flags). `ApiClient` gains `get_repo_config` (GET `/api/v1/onboarding/repos/{repo}`) and `upsert_repo_config` (POST `/api/v1/onboarding/repos`). 404 on GET surfaces "run `treadmill onboarding add` first"; 422 on POST surfaces the validation error with exit 1.
