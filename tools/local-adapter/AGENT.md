# tools/local-adapter

## Purpose

This directory contains the Treadmill-native local adapter, the bridge between CDK synth output and a moto + native Docker substrate for local-first development. It reads the CloudFormation JSON synthesized by the CDK app and operationalizes it by provisioning AWS-managed primitives (SNS, SQS, S3, IAM, Secrets, SSM) against moto, launching Postgres and Redis as native containers, wiring ECS task definitions to docker run commands, and running an autoscaling control loop that matches SQS queue depth to container count. The adapter ensures local behavior is faithful to AWS production for the cases developers care about: service definitions, scaling policies, credential injection, and the exit-then-restart cycle.

## Key surfaces

- `treadmill_local/cli.py` — entry point; `treadmill-local up` brings the substrate online, `down` tears it down, `logs` streams container output. The `repo onboard` subcommand registers the cwd's repo with the running deployment per ADR-0051.
- `treadmill_local/runtime.py` — core orchestrator; loads CDK JSON, dispatches to provisioners, starts containers, runs the autoscaler loop. `_run_container` accepts an optional `docker_adapter` (ADR-0060): when provided, worker containers are spawned on the `treadmill-egress` network with proxy env vars and per-worker install credentials injected.
- `treadmill_local/provisioner.py` — provisions AWS primitives (SNS, SQS, S3, IAM, Secrets, SSM) into moto.
- `treadmill_local/autoscaler.py` — target-tracking control loop; reads SQS depth from moto, docker stats from Docker daemon, computes desired worker count from scaling policy, launches/drains containers. On startup, ensures the `treadmill-egress` network and proxy container exist (ADR-0060); passes `DockerClientAdapter` to each worker spawn for egress isolation.
- `treadmill_local/docker_client.py` — thin `DockerClientAdapter` wrapping docker-py for dependency injection; all egress-proxy Docker interactions go through this adapter so tests can substitute a fake without a real daemon.
- `treadmill_local/egress_proxy.py` — egress proxy lifecycle helpers (ADR-0060): `ensure_egress_network`, `ensure_egress_proxy_container` (idempotent), `mint_worker_credential`, `build_always_allowed`, `build_install_allowed`, `write_worker_allowlist`.
- `treadmill_local/deployment_config.py` — writes `~/.treadmill/<deployment_id>.yaml` with container ports and endpoint URLs per ADR-0016's schema.
- `treadmill_local/subprocess_logging.py` — bounded logging helpers for the autoscaler + deploy-watcher subprocesses: a `configure_rotating_logging` setup (size-rotating file handler replacing stdout) and a `RateLimitedErrorLogger` that logs the first occurrence of an error in full and collapses repeats into periodic counted summaries. The scheduler subprocess uses the parallel implementation in `services/api/treadmill_api/scheduler/bounded_logging.py` (separate package; same shape but not shared by import).
- `treadmill_local/onboard.py` — pure helpers for client-side onboarding discovery (ADR-0051): `infer_repo` parses git remotes, `build_profile` produces a minimal `repo_profile` from the local checkout, `onboard_payload` assembles the POST body.
- `treadmill_local/docs_sync.py` — pure, testable doc sync helpers (ADR-0054): `list_docs`, `get_doc`, `pull`, `push`. Each function accepts injected HTTP callables so tests mock entirely without a network. `pull` downloads all docs from the API into a local directory; `push` uploads all files under a local directory to the API (last-write-wins, no-op when the dir is absent). The `docs` CLI command group wires these into `list` / `get` / `pull` / `push` subcommands, and **scopes the local mirror by repo**: `--dir` is a root and files land under `<dir>/<owner>/<name>/<doc_path>`, so multiple projects don't collide locally (mirrors the S3 `repo-context/<repo>/` layout).

## Recent changes

- PR — ADR-0060 Step 3b: autoscaler now ensures a `treadmill-egress` internal bridge network and spawns the `treadmill-egress-proxy` container on startup. Each worker is spawned on that network (no external gateway) with `HTTP_PROXY`/`HTTPS_PROXY`/`TREADMILL_INSTALL_PROXY_TOKEN` env vars injected. A per-worker `WorkerAllowlist` JSON is written to `<infra_dir>/egress-proxy-config/<worker_ip>.json` after the IP is resolved. New modules: `docker_client.py` (adapter) and `egress_proxy.py` (lifecycle helpers). `start_worker_once` and `_run_container` in `runtime.py` accept an optional `docker_adapter` kwarg for this path.
- PR — ADR-0056 dashboard auto-deploy. `runtime.py` gains `recreate_dashboard_container` (mirror of `recreate_api_container`): runs `_ensure_images_built` for the shared rebuild path, force-removes the running `treadmill-dashboard` container, and `docker run`s a new one from the freshly-tagged `treadmill-dashboard:dev` using `_build_dashboard_service_spec` so `up` and the recreate path share one creation site. `deploy_watcher.py` gains a `services/dashboard/` → `dashboard` entry in `_DISPATCH_TABLE` (first-match wins; sits between `services/api/` and `workers/agent/`), a `_action_dashboard` action that `_sync_local_to_origin` + invokes the recreate fn, and a required `recreate_dashboard_fn` kwarg on `DeployWatcher.__init__` wired through both the dev-local and fully-local CLI entrypoints. No `docker build` shell-out from the watcher and no `/health/ready` probe — the dashboard is static nginx, the build path is owned by the runtime helper.
- PR — ADR-0053 Wave 3: `_dev_local_worker_env` now lifts `aws.corpus_s3_uri` (OPERATOR-supplied YAML field, not a CDK output) into the agent worker's `TREADMILL_CORPUS_S3_URI` env var, so `tools/load-analysis-corpus.sh pull` works from inside the worker. Absent in YAML → env var omitted entirely (the optimizer surfaces a clean configuration error rather than failing deep in boto3).
- PR — `docs pull`/`push` now scope the local mirror by repo: the effective dir is `<--dir>/<owner>/<name>/`, matching the S3 `repo-context/<repo>/` key layout so pulling several projects into one root no longer collides. `push` is a no-op when the repo-scoped subdir is absent (e.g. before first author) instead of relying on `rglob` over a missing path.
- ADR-0055 step 4 — operator surface. `build_deployment_config` (`deployment_config.py`) scans CFN outputs for values matching `treadmill-<id>/claude-account-<name>` and emits one `aws.claude_accounts` entry per — `{name, type: "oauth", secret_name}` — sorted by name. `claude_default_account` is left unset so the operator picks which account is default. `runtime.py` reads `aws.claude_accounts` from the operator YAML, serialises it as `CLAUDE_ACCOUNTS_JSON` (`{name: {type, secret_name}}`), and injects that plus `CLAUDE_DEFAULT_ACCOUNT` into the API container env. Empty/absent ⇒ API's resolver returns 503 and the worker keeps using the existing `CLAUDE_CREDENTIALS_PATH` bind-mount (no behavior change).
- PR — Closed the stale-source sibling to ADR-0024. `_action_api` and `_action_agent` now `git fetch origin` + `git merge --ff-only origin/main` in the local clone before each `docker build`, via a new `DeployWatcher._sync_local_to_origin` helper. Without this, `docker build <local_dir>` packaged whatever was on disk — so when the operator's clone was behind `origin/main` the "fresh" image was built from pre-merge source (the 2026-05-26 OTel canary observed exactly this: the gRPC→HTTP exporter fix merged but the rebuilt API image still contained the gRPC client). ff-only failure (unpushed local commits, different branch checked out) logs a warning that includes the merge stderr and falls back to building from current local state — preserving today's behavior so the sync step doesn't introduce a new silent-skip vector. All git calls are plain `subprocess.run` with `check=False` (no GitPython dep).
- PR — Fixed deploy-watcher's silent-no-op auto-deploy for the API (ADR-0024). `deploy_watcher._action_api` now (a) `docker build`s the fresh `treadmill-api:dev` image and then (b) **recreates** the container from it via a new `LocalRuntime.recreate_api_container` helper — force-removing the running container and `docker run`ning a new one with the same env/ports/network as `up`. The previous `docker restart` re-ran the EXISTING container's image, so the freshly-built bits (api code + migrations) never went live. Recreate goes through `_build_api_service_spec` (factored out of `_build_dev_local_service_specs`) so `up` and the deploy-watcher share one creation path and can't drift. The post-deploy health probe now targets the dev-local API port from the deployment YAML (`cfg["local"]["api_url"]` → `http://localhost:8088/health/ready`) instead of the hardcoded `:8000` that silently timed out against the wrong port; the YAML default in `_LOCAL_DEFAULTS` is corrected from `:8000` to `:8088` to match the runtime's `DEV_LOCAL_API_HOST_PORT` constant.
- PR — Added `treadmill-local docs` command group (ADR-0054): `list`, `get`, `pull`, `push` subcommands backed by a new `docs_sync.py` module. Functions take injected HTTP callables for testability. `pull` writes all docs to `.treadmill-docs/<doc_path>` (mkdir parents); `push` PUTs each file with `{"content": ...}` (last-write-wins). Repo is inferred from cwd git remote when `--repo` is omitted.

## Pitfalls

- Moto does not implement all AWS service behaviors; changes to the API's SQS or SNS usage can fail locally but succeed against real AWS. Always validate changes against a real AWS environment before moving to production.
- Docker resource limits (memory, CPU) in adapter container specs are hints, not enforced; if containers exceed their limits locally, Docker may kill them without warning. Use `docker stats` to monitor and test load locally before deploying.
- The autoscaler reads moto's SQS state directly; if moto's SQS implementation diverges from AWS behavior (message visibility, batch operations, long polling), the autoscaler may misbehave. Monitor the adapter's logs for scale-up/down decisions during development.
- Host-side credential injection via `startup_auth.py` happens at adapter startup; if credentials rotate or become invalid mid-session, the running containers will not refresh them until the next `treadmill-local up`.
