# tools/local-adapter

## Purpose

This directory contains the Treadmill-native local adapter, the bridge between CDK synth output and a moto + native Docker substrate for local-first development. It reads the CloudFormation JSON synthesized by the CDK app and operationalizes it by provisioning AWS-managed primitives (SNS, SQS, S3, IAM, Secrets, SSM) against moto, launching Postgres and Redis as native containers, wiring ECS task definitions to docker run commands, and running an autoscaling control loop that matches SQS queue depth to container count. The adapter ensures local behavior is faithful to AWS production for the cases developers care about: service definitions, scaling policies, credential injection, and the exit-then-restart cycle.

## Key surfaces

- `treadmill_local/cli.py` — entry point; `treadmill-local up` brings the substrate online, `down` tears it down, `logs` streams container output. The `repo onboard` subcommand registers the cwd's repo with the running deployment per ADR-0051.
- `treadmill_local/runtime.py` — core orchestrator; loads CDK JSON, dispatches to provisioners, starts containers, runs the autoscaler loop. `_run_container` accepts an optional `docker_adapter` (ADR-0060): when provided, worker containers are spawned on the `treadmill-egress` network with proxy env vars and per-worker install credentials injected. `_up_dev_local` calls `ensure_egress_network` (ADR-0064 Step 1) before `_start_services` so the egress network exists when the API spawns and can be multi-attached to it. `_start_services` + `recreate_api_container` then call `_multi_attach_api_to_egress` after the API container is up (ADR-0064 Step 2) so workers on the internal-only egress bridge can resolve `treadmill-api` by DNS.
- `treadmill_local/provisioner.py` — provisions AWS primitives (SNS, SQS, S3, IAM, Secrets, SSM) into moto.
- `treadmill_local/autoscaler.py` — target-tracking control loop; reads SQS depth from moto, docker stats from Docker daemon, computes desired worker count from scaling policy, launches/drains containers. On startup, ensures the `treadmill-egress` network and proxy container exist (ADR-0060); passes `DockerClientAdapter` to each worker spawn for egress isolation.
- `treadmill_local/docker_client.py` — thin `DockerClientAdapter` wrapping docker-py for dependency injection; all egress-proxy Docker interactions go through this adapter so tests can substitute a fake without a real daemon. Includes `connect_container_to_network` (ADR-0064 Step 2) for multi-attaching an already-running container to a second network (idempotent — no-op when already attached).
- `treadmill_local/egress_proxy.py` — egress proxy lifecycle helpers (ADR-0060): `ensure_egress_network`, `ensure_egress_proxy_container` (idempotent — multi-attaches the proxy to `treadmill-local` per ADR-0064 Step 2 so outbound CONNECTs route through that network's gateway), `mint_worker_credential`, `build_always_allowed`, `build_install_allowed`, `write_worker_allowlist`.
- `treadmill_local/deployment_config.py` — writes `~/.treadmill/<deployment_id>.yaml` with container ports and endpoint URLs per ADR-0016's schema.
- `treadmill_local/subprocess_logging.py` — bounded logging helpers for the autoscaler + deploy-watcher subprocesses: a `configure_rotating_logging` setup (size-rotating file handler replacing stdout) and a `RateLimitedErrorLogger` that logs the first occurrence of an error in full and collapses repeats into periodic counted summaries. The scheduler subprocess uses the parallel implementation in `services/api/treadmill_api/scheduler/bounded_logging.py` (separate package; same shape but not shared by import).
- `treadmill_local/onboard.py` — pure helpers for client-side onboarding discovery (ADR-0051): `infer_repo` parses git remotes, `build_profile` produces a minimal `repo_profile` from the local checkout, `onboard_payload` assembles the POST body.
- `treadmill_local/docs_sync.py` — pure, testable doc sync helpers (ADR-0054): `list_docs`, `get_doc`, `pull`, `push`. Each function accepts injected HTTP callables so tests mock entirely without a network. `pull` downloads all docs from the API into a local directory; `push` uploads all files under a local directory to the API (last-write-wins, no-op when the dir is absent). The `docs` CLI command group wires these into `list` / `get` / `pull` / `push` subcommands, and **scopes the local mirror by repo**: `--dir` is a root and files land under `<dir>/<owner>/<name>/<doc_path>`, so multiple projects don't collide locally (mirrors the S3 `repo-context/<repo>/` layout).

The proper fix is multi-attach the API to both networks (or move all internal services onto the egress network) plus a real end-to-end smoke test. Until that lands, the autoscaler checks `TREADMILL_EGRESS_PROXY_ENABLED` (default `false`); when off, workers spawn on the operator-default network like pre-#92 — no proxy, no per-worker isolation, but they actually run. - PR — ADR-0060 Step 3b follow-up #2: workers now get `NO_PROXY=treadmill-api,localhost,127.0.0.1` in their spawn env so internal Treadmill-API calls bypass the egress proxy. ## Pitfalls

- Moto does not implement all AWS service behaviors; changes to the API's SQS or SNS usage can fail locally but succeed against real AWS. Always validate changes against a real AWS environment before moving to production.
- Docker resource limits (memory, CPU) in adapter container specs are hints, not enforced; if containers exceed their limits locally, Docker may kill them without warning. Use `docker stats` to monitor and test load locally before deploying.
- The autoscaler reads moto's SQS state directly; if moto's SQS implementation diverges from AWS behavior (message visibility, batch operations, long polling), the autoscaler may misbehave. Monitor the adapter's logs for scale-up/down decisions during development.
- Host-side credential injection via `startup_auth.py` happens at adapter startup; if credentials rotate or become invalid mid-session, the running containers will not refresh them until the next `treadmill-local up`.
