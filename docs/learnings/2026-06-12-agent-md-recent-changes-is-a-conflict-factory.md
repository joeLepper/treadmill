# AGENT.md Recent-changes prepend is a conflict factory → per-PR fragments

**Date:** 2026-06-12
**Trigger:** task 986c5cf6, after three same-day rework cascades.

## Incident

Every PR prepends its Recent-changes entry at the same anchor — the
line after `## Recent changes` in the component's AGENT.md. With more
than one PR in flight, every merge re-conflicts every open PR at that
anchor. On 2026-06-12 alone, `services/api/AGENT.md` produced three
cascades (PR #341 vs #343/#344, then again vs #339), each resolved
keep-both by hand, each costing a coordinator rework dispatch, a merge,
suites, and a push. The resolution was identical and deterministic all
three times — a textbook automatable conflict, generated by convention.

## Incumbents checked (the /decide rule)

1. **`docs/agent-md-schema.md`** — pins `Recent changes` as a required
   AGENT.md section ("last three notable changes with PR links"). The
   section must survive; only its write path is the problem.
2. **`workers/agent/treadmill_agent/conflict_preresolve.py`** (triage
   71ed396b, PR #144) — the repo already built a deterministic
   keep-both resolver for exactly this class ("additive-list-head"
   conflicts), guarded by four structural conditions (purely additive,
   list-shaped lines, pre/post anchor checks). It is wired only into
   the retired ADR-0018 container runner, so the live team flow has no
   pre-resolution — the mitigation died with its host. Two judgments
   carry forward: (a) the class is fully deterministic, and (b) blind
   both-keep was NOT considered safe without structural guards.
3. **`docs/learnings/YYYY-MM-DD-*.md`** — the repo's established
   dated-fragment-file pattern: concurrent additions never conflict
   because each addition is its own file.

## Decision: per-PR fragment files

New Recent-changes entries are files, not prepends:

    <component>/agent-changes/YYYY-MM-DD-<task-or-pr-slug>.md

One entry per file, same content shape as before (what changed, PR
link). Newest-first falls out of filename sort. The `## Recent
changes` section in each AGENT.md becomes a pointer to the directory;
entries that predate the convention stay frozen in place until
gardening folds them into the prose sections above. Gardening a
fragment = fold the load-bearing fact into Purpose/Key
surfaces/Pitfalls and delete the file — a deletion, which cannot
conflict with concurrent additions.

This makes the conflict structurally impossible rather than
auto-resolved: new files never collide, so no resolver — alive, dead,
or gitattribute — is needed at all.

## Why not the alternatives

- **`merge=union` gitattribute:** applies file-wide with no
  guardrails. The repo's own #144 prior art encodes the judgment that
  blind both-keep needs four structural conditions a gitattribute
  cannot express. Union silently both-keeps genuine prose conflicts in
  the other four schema sections (Purpose/Key surfaces/Pitfalls/
  Navigation), can resurrect lines deleted by gardening when a prune
  races an add, and makes entry order merge-order-dependent. Worst of
  all, the corruption is silent in files whose consumers are LLM
  sessions reading them as ground truth — nobody sees the garble.
- **Append-under-dated-headings:** does not address the observed
  class. All three 2026-06-12 cascades were same-day; they would share
  the dated heading's anchor and conflict identically. Date
  granularity relocates the anchor; it does not remove it.

## Enforcement

`tools/dev-hooks/tests/test_agent_md_conventions.py` walks the repo:
every AGENT.md carrying `## Recent changes` must carry the
fragment-pointer text, and every `agent-changes/` entry must match the
`YYYY-MM-DD-*.md` shape. `docs/agent-md-schema.md` defines the
authoring flow.
